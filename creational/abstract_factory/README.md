# 抽象工厂模式 (Abstract Factory Pattern)

## 概述

抽象工厂模式用于创建一组相关或互相依赖的对象族，而不需要显式指定它们的具体类型。客户端只依赖抽象工厂和抽象产品接口，因此可以在不同产品族之间切换。

## Go 语言实现特点

- 通过接口定义产品族和工厂契约
- 客户端只依赖接口，不依赖具体实现
- 常与依赖注入、组合一起使用
- 比起继承层次，更适合使用小接口拆分能力

## 核心概念

```go
type Button interface {
    Render() string
}

type Checkbox interface {
    Render() string
}

type UIFactory interface {
    CreateButton() Button
    CreateCheckbox() Checkbox
}
```

## 代码示例

当前实现模拟 UI 主题工厂：

- `MacFactory` 生成 `MacButton` 和 `MacCheckbox`
- `WinFactory` 生成 `WinButton` 和 `WinCheckbox`

客户端只依赖 `UIFactory`，即可在不同主题之间切换而不修改业务代码。

## 为什么 Go 中较少使用这种重量级工厂

抽象工厂在 Java/C++ 一类语言中很常见，因为这些语言经常围绕“类族 + 继承层次”构建对象系统。但在 Go 中，它并不是高频模式，原因主要有：

1. Go 更推崇组合和小接口，而不是庞大的类型族
2. 很多创建逻辑用简单构造函数就足够表达
3. 依赖注入在 Go 中通常直接传接口实现，无需再套一层重量级工厂体系
4. 如果产品族不复杂，引入抽象工厂反而会增加样板代码

因此，在 Go 中更常见的做法往往是：

- 构造函数
- 工厂函数
- 函数式选项
- 直接注入接口实现

只有当“产品族必须整体切换，且需要保证一致性”时，抽象工厂才真正划算。

## 使用场景

1. 一组对象必须成套创建
2. 需要在多个产品族之间切换
3. 希望客户端完全隔离具体产品实现

## 优缺点

### 优点

- 保证产品族一致性
- 隔离具体实现
- 切换产品族方便

### 缺点

- 样板代码较多
- 类型层次更重
- 在 Go 中常常不如简单工厂直接

## 与其他模式的关系

- 与 **工厂方法** 相比：工厂方法通常只负责一种产品，抽象工厂负责一整族产品
- 与 **建造者模式** 相比：建造者关注复杂对象逐步构建，抽象工厂关注成组对象选择
